# 트랜잭션

트랜잭션은 Arweave 네트워크의 기본 구성 요소입니다.

이 섹션에서는 서로 다른 유형의 트랜잭션을 설명하고 네트워크에 게시하는 방법들을 소개합니다.

## 트랜잭션 게시

대부분의 블록체인과 마찬가지로, Arweave는 트랜잭션을 생성하고 서명한 뒤 네트워크에 게시해야 합니다.

Arweave에는 트랜잭션을 게시하는 여러 방법이 있으며, 일반적으로 [gateway]()로 직접 게시하는 것, [bundler]()를 사용하는 것, 또는 디스패치(dispatch)하는 방식으로 구분됩니다.

## 트랜잭션 유형

Arweave에 직접 게시되는 트랜잭션은 두 가지 유형이 있습니다: **지갑 간(wallet-to-wallet)** 트랜잭션과 **데이터(data)** 트랜잭션. 첫 번째는 지갑 주소 간에 **AR** 토큰을 전송합니다. 두 번째는 데이터를 Arweave에 게시하고 관련된 저장 비용을 지불합니다.

흥미롭게도, **데이터** 트랜잭션은 저장 비용을 지불하는 동시에 지갑 주소로 **AR** 토큰을 전송할 수도 있습니다.

모든 트랜잭션은 사용자가 [custom tags](./tags.md) 형태로 최대 2KB 분량의 메타데이터를 지정할 수 있도록 합니다.

### 게이트웨이로 직접 게시

게이트웨이(gateway)는 클라이언트와 Arweave 피어 네트워크 사이에 위치합니다. 게이트웨이의 주요 기능 중 하나는 트랜잭션을 인덱싱하고 블록에 포함될 때까지 네트워크에 게시된 데이터를 낙관적으로 캐싱하는 것입니다. 이로 인해 트랜잭션은 거의 즉시 "Pending" 상태로 질의 가능해지며, 게이트웨이를 기반으로 구축된 애플리케이션이 더 반응성 있게 동작할 수 있습니다. 다만 피어들이 블록에 트랜잭션을 채굴하지 않으면 낙관적 캐시에서 트랜잭션이 사라질 위험이 남아 있습니다.

`arweave-js`를 사용하여 직접 트랜잭션을 게시하는 예시는 [이 가이드](../../guides/posting-transactions/arweave-js)에서 확인할 수 있습니다.

::: tip 보장된 트랜잭션
대량의 트랜잭션을 게시하거나 빠른 정산 시간이 필요할 때는 번들링 서비스(bundling service)를 사용하는 것을 고려하세요.

번들러는 대량의 트랜잭션을 즉시 정산하고 트랜잭션 데이터를 밀리초 단위로 이용 가능하게 합니다. 번들링 서비스는 게시된 트랜잭션을 온체인에서 확인될 때까지 보관합니다.

트랜잭션이 최신 블록에 포함되지 않으면 번들링 서비스는 각 새 블록마다 트랜잭션을 재게시하여 충분한 확인 수로 온체인에 기록될 때까지 처리합니다.
:::

## 번들된 트랜잭션

Permaweb 빌더를 위한 추가 유틸리티를 제공하는 Arweave 위에 구축된 서비스들은 때때로 Permaweb 서비스라고 불립니다. 번들러는 그러한 서비스 중 하나입니다.

번들러는 여러 개의 개별 트랜잭션을 모아 하나의 트랜잭션으로 묶어 Arweave에 직접 게시합니다.

이렇게 프로토콜 수준의 단일 트랜잭션은 수만 개의 번들된 트랜잭션을 포함할 수 있습니다. 단, 한 가지 제한이 있는데, 번들에 포함될 수 있는 것은 오직 **데이터** 트랜잭션뿐이라는 점입니다. 지갑 주소 간 **AR** 토큰을 전송하는 **지갑 간(wallet-to-wallet)** 트랜잭션은 개별 트랜잭션으로 Arweave에 직접 게시되어야 합니다.

## 디스패치된 트랜잭션

또 다른 번들된 트랜잭션 게시 방법은 브라우저에서입니다. 브라우저는 업로드할 수 있는 데이터 크기에 일부 제약을 가하지만, 브라우저 기반 지갑은 번들러에 트랜잭션을 게시할 수 있습니다.

Arweave 브라우저 지갑은 `dispatch()` API 메서드를 구현합니다. 만약 100KB 이하의 작은 트랜잭션을 게시하는 경우, 지갑의 `dispatch()` 메서드를 사용하여 번들된 트랜잭션의 이점을 활용할 수 있습니다.

Arweave 지갑의 `dispatch()` 메서드로 100KB 이하의 번들된 트랜잭션을 게시하는 예시는 [이 가이드](../../guides/posting-transactions/dispatch.md)에서 확인할 수 있습니다.
